# Product Requirements Document (PRD)

## �� 프로젝트 개요
### 프로젝트명
**Soup** - 스터디 운영 및 학습 관리 플랫폼

### 목표
- **학습 의욕 증진**: 사용자가 내가 해온 학습 기록을 한눈에 확인할 수 있도록 함
- **사용자 편의**: 중요 일정과 회고 제출 등의 미션을 잊지 않도록 알림 제공
- **운영자 편의**: 스터디 운영자가 구성원 관리 및 일정 운영을 편하게 할 수 있도록 함

### 범위
- **포함**: 웹 기반 스터디 관리, 회고/심화학습 제출, 알림 시스템, 모바일 대응
- **제외**: 별도 앱 설치 없이 웹 기반으로 구현

## 🏗️ 아키텍처 원칙
### 기술 스택 선택 이유
- **Spring Boot**: 회고 제출 등 일반적인 CRUD 작업에 적합한 프레임워크
- **Database**: 복잡한 스터디-일정-섹션-유저 관계를 효율적으로 관리
- **Frontend**: 반응형 웹 기반, PWA 적용 고려 (앱처럼 사용 가능)
- **Quartz Scheduler**: 정기적 푸시 알림 (예: 매일 아침 9시 전날 회고 누락자 알림)

### 설계 원칙
- **일정 중심 UI**: 회고/심화 흐름 강조
- **데이터 구조**: Materialized Path 방식으로 복잡한 관계 관리
- **성능 최적화**: Redis 캐싱, 피벗 테이블 자동 갱신 고려

## 📚 용어 정의
- **스터디**: 학습 그룹 단위
- **섹션**: 강의 단위 (스터디 내 구성 요소)
- **일정**: 여러 섹션을 포함하는 시간 단위
- **회고**: 각 섹션에 대한 학습 후기
- **심화학습**: 일정별로 제출하는 심화 학습 자료

## 🎯 핵심 기능 요구사항

### 기능 1: 스터디 구성 관리
- **목적**: 스터디 운영자가 섹션과 일정을 체계적으로 관리
- **입력**: 스터디 생성, 섹션 추가, 일정 설정
- **출력**: 스터디 구조 (스터디 → 일정 → 섹션 계층)
- **비즈니스 로직**: 
  - 한 개의 스터디는 여러 개의 섹션과 일정으로 구성
  - 하나의 일정은 여러 섹션을 포함 가능
  - 보통 섹션 하나당 하나의 회고, 하나의 일정에 하나의 심화학습 배정
- **예외 상황**: 섹션이 일정에 할당되지 않은 경우

### 기능 2: 회고/심화학습 제출
- **목적**: 참여자가 학습 내용을 체계적으로 기록하고 제출
- **입력**: 회고 내용, 심화학습 자료
- **출력**: 제출 현황, 완료율
- **비즈니스 로직**:
  - 각 일정에 포함된 섹션들에 대해 회고 작성
  - 일정별로 심화 학습 자료 제출
  - 제출 여부 추적 및 통계
- **예외 상황**: 제출 기한 초과, 중복 제출 방지

### 기능 3: 캘린더 기반 일정 관리
- **목적**: 사용자가 다가오는 일정을 직관적으로 확인
- **입력**: 일정 데이터, 제출 현황
- **출력**: 캘린더 뷰, 제출 여부 색깔 표시
- **비즈니스 로직**: 
  - 캘린더 기반으로 다가오는 일정 표시
  - 제출 완료/미완료 상태를 색깔로 구분
  - 전체 회고/심화 학습 현황 확인
- **예외 상황**: 일정 변경 시 캘린더 동기화

### 기능 4: 알림 시스템
- **목적**: 중요 일정과 미션을 잊지 않도록 알림 제공
- **입력**: 일정 정보, 제출 현황, 사용자 설정
- **출력**: 푸시 알림, 이메일 알림
- **비즈니스 로직**:
  - 매일 아침 9시 전날 회고 누락자 알림
  - 다가오는 일정 알림
  - 제출 기한 임박 알림
- **예외 상황**: 알림 설정 비활성화, 알림 전송 실패

### 기능 5: 관리자 대시보드
- **목적**: 스터디 운영자가 구성원 관리 및 운영 현황 파악
- **입력**: 스터디 데이터, 제출 현황, 사용자 정보
- **출력**: 통계 리포트, 관리 도구
- **비즈니스 로직**:
  - "이번 주에 심화학습을 안 낸 유저 누구야?"
  - "스터디별로 회고 미제출 섹션 수를 보여줘"
  - 구성원별 참여도 분석
- **예외 상황**: 데이터 없음, 권한 부족

## 📊 데이터 모델
### 핵심 엔티티 관계
```
[스터디] 안에 있는
  └ [일정] 중 하나에서
     └ 포함된 [섹션] 각각에 대해
        └ [유저]가 회고를 했는지 여부
     └ 그 [일정]에 대해
        └ [유저]가 심화학습을 냈는지 여부
```

### 데이터 관리 전략
1. **Materialized Path**: `study/1/schedule/3/section/2/user/17` + Redis 캐싱
2. **피벗 테이블 자동 갱신**: 통계 데이터 실시간 업데이트
3. **쿼리 + 인덱싱**: 복잡한 조회 성능 최적화
4. **배치 처리**: 대용량 데이터 처리

## 🔄 API 설계 원칙
- **RESTful API**: 표준 HTTP 메서드 사용
- **일정 중심 설계**: 일정을 기준으로 한 리소스 구조
- **상태 추적**: 제출 여부를 명확히 표시하는 응답 구조
- **페이지네이션**: 대용량 데이터 처리

## 🧪 테스트 전략
- **단위 테스트**: 각 서비스 레이어별 테스트
- **통합 테스트**: API 엔드포인트 테스트
- **성능 테스트**: 복잡한 조회 쿼리 성능 검증
- **알림 테스트**: 스케줄러 기반 알림 기능 검증
